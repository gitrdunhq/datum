# Phase: Architect Gate

**Goal:** Validate architectural decisions, audit ADR coverage, and enforce C4 context mapping before any code is written.

## Inputs
- `TASKS.md` — implementation plan
- `docs/PROPERTIES.md` — invariants
- `docs/ARCHITECTURE.md` (if present)
- `docs/adr/` — existing architecture decision records

## Protocol

This gate executes after `02-plan.md` and before `04-act.md`.

### 1. Identify Significant Decisions
A decision from `TASKS.md` is "significant" if it:
- Introduces a new dependency, framework, or external service
- Changes a module boundary, public API, or inter-service contract
- Alters data persistence, schema, or migration strategy
- Affects security, auth, or access control
- Establishes a new architectural pattern

### 2. ADR Audit
For each significant decision:
1. Check `docs/adr/` for an existing, current ADR that covers it.
2. If covered and current, proceed.
3. If missing or stale, MUST write a new ADR using the MADR format to `docs/adr/`.
   - Title: `NNN-<kebab-slug>.md`
   - Status: `Accepted`
   - Context: why this decision arose
   - Decision: what was decided
   - Consequences: what this enables and constrains

### 3. C4 Diagram Updates
Identify structural changes to components or dependencies.
Update `docs/ARCHITECTURE.md` to reflect these changes using Mermaid C4 block diagrams. Ensure new components have clear boundaries and explicitly documented interfaces.

### 4. Validation Verdict
If the plan introduces severe coupling, violates layer boundaries, or introduces unacceptable dependency risk without an explicit mitigation in the ADR, the Architect Gate MUST halt and escalate to the user with a `validation_failed` status.

## Outputs
- Updated `docs/ARCHITECTURE.md`
- New or updated MADR files in `docs/adr/`
- `architect-output.json` containing the verdict and ADR inventory.

Do not proceed to ACT until this gate passes.
