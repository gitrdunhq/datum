# Phase: Research (R&D Department)

**Goal:** Establish a strict boundary for high-uncertainty exploration, prior-art comparison, and architectural spikes *before* entering the Plan phase.

## Core Contract

```text
Research investigates.
Architecture decides.
Implementation waits for promotion.
```

The Research phase explores options and proposes solutions without committing code. Agents in this phase **must not** implement tests, schemas, tools, or production logic.

## Inputs

- `docs/epics/<branch>/SPEC.md`
- Current documentation and codebase context
- External API/tool docs (via Context7 MCP if needed)

## Steps

### 1. Identify Research Need
If the `SPEC.md` involves high uncertainty (e.g., choosing a database, designing a complex manager view, or evaluating a new framework), the Epic must pass through the Research phase.

### 2. Conduct Research & Spikes
Agents generate options, compare prior art, and draft Architecture Decision Records (ADRs).
- **Allowed:** Writing memos, updating `docs/research/`, proposing roadmap slots, or creating ADRs.
- **Forbidden:** Creating `TASKS.md`, generating application source code, or mutating the primary runtime logic.

### 3. Promotion Gate
Research outputs must be explicitly promoted to the Plan phase by a human manager.

- Run `datum gate research [--yolo]`
- The gate validates that no runtime paths were modified and that an explicit `recommendation` or `ADR` was produced.
- The manager signs off to promote the Epic into `02-plan.md`.

## Outputs

- `docs/research/<branch>-findings.md`
- Draft `docs/adr/` records
- Validated assumptions to feed into `02-plan.md`
