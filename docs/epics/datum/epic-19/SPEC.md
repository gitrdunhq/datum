# Spec: Auto-generate walkthrough.md summaries

**Run ID:** <!-- filled by datum -->
**Phase:** Refine
**Status:** Draft

---

## 1. Summary

Add auto-generation of a `WALKTHROUGH.md` post-mortem artifact at the end of the `datum closeout` command. Generation uses local LLM (Gemma via `sidecar_docs` phase) to summarize git diff, SPEC.md, and TASKS.md into a human-readable technical narrative. A `WALKTHROUGH.md` template is added to `templates/` and the generation is integrated into the closeout render pipeline.

## 2. Context

Currently agents manually write walkthrough summaries at the end of an epic (e.g., "schemas updated, ACI execution loop added"). This is inconsistent — some epics get detailed walkthroughs, others get none. The issue asks for an automated artifact, templated so it's uniform, generated by local LLM so it's intelligent (not just a git log dump), and stored in `docs/epics/<branch>/WALKTHROUGH.md` alongside SPEC, TASKS, and PROPERTIES.

The `sidecar_docs` phase in `local_llm.py` is the correct routing key — it already exists in the phases list and is semantically correct for "generate documentation from code changes".

## 3. Requirements

### R1: WALKTHROUGH.md template in templates/

**Description:** Add `templates/WALKTHROUGH.md` as a Jinja2-style or hand-filled template that defines the structure of a walkthrough artifact. The template must include: summary of changes, technical narrative by lane (RED/GREEN/REFACTOR), files touched, key design decisions, and what was explicitly excluded.

**Acceptance criteria:**
- AC1: `templates/WALKTHROUGH.md` exists and has all required sections
- AC2: Each section has a clear placeholder comment explaining what to fill
- AC3: The template can be filled by local LLM output with no structural changes

### R2: `datum walkthrough` subcommand generates the artifact

**Description:** A new `datum walkthrough` CLI subcommand (also invoked at the end of `datum closeout`) reads the current epic's SPEC.md, TASKS.md, and `git diff main..HEAD` and passes them to `run_phase("sidecar_docs", ...)` to produce a WalkthroughSummary. The summary is then rendered into `docs/epics/<branch>/WALKTHROUGH.md` using the template.

**Acceptance criteria:**
- AC1: `datum walkthrough` runs without error on an epic with SPEC.md + TASKS.md
- AC2: Output file written to `docs/epics/<branch>/WALKTHROUGH.md`
- AC3: If `sidecar_docs` local LLM phase is unavailable, produce a deterministic fallback from TASKS.md alone (no Claude escalation)
- AC4: `datum closeout` calls `datum walkthrough` after archiving artifacts

### R3: WalkthroughSummary Pydantic schema

**Description:** A Pydantic model `WalkthroughSummary` defines the structured output that Gemma writes. Fields: `summary` (str), `lanes` (list of lane narratives), `files_touched` (list[str]), `key_decisions` (list[str]), `excluded` (list[str]).

**Acceptance criteria:**
- AC1: `datum/models/walkthrough_schema.py` defines `WalkthroughSummary`
- AC2: Schema validates cleanly with Pydantic v2
- AC3: `structured()` call in `local_llm.py` can use this schema

### R4: Unit and integration tests

**Description:** Tests cover: template exists, schema validates, deterministic fallback works, subcommand exits 0 on a fixture epic dir, and `run_phase("sidecar_docs", ...)` is called with the correct prompt.

**Acceptance criteria:**
- AC1: `tests/test_walkthrough.py` — minimum 5 tests covering the above
- AC2: All tests pass with `uv run pytest tests/test_walkthrough.py`
- AC3: Tests do NOT call Gemma (mock `run_phase`)

## 4. Failure Modes and Handling

| Failure | Handling |
|---|---|
| `sidecar_docs` not in local_llm phases | Fall back to deterministic render from TASKS.md — no Claude escalation |
| Gemma returns malformed JSON | Catch Pydantic validation error, log warning, write deterministic fallback |
| SPEC.md or TASKS.md missing | Log warning, use available inputs; if both missing, write empty template shell |
| `git diff` empty (no commits) | Omit diff section from prompt; summarize from TASKS.md only |
| Output dir doesn't exist | Create `docs/epics/<branch>/` with `mkdir -p` before writing |

## 5. Non-Functional Requirements

| Requirement | Target |
|---|---|
| Generation time (Gemma) | < 60s for a typical epic (< 5K token prompt) |
| Output size | < 500 lines |
| Deterministic fallback | Always produces valid markdown, even with empty inputs |
| No new runtime dependencies | Uses existing `mlx-community/gemma-4-26b-a4b-it-4bit` and Pydantic |

## 6. Out of Scope

- Walkthrough generation for individual pipeline phases (only epic-level)
- Rendering walkthrough as HTML or PDF
- Uploading walkthrough to GitHub PR body automatically
- Semantic diffing against prior walkthroughs
- Using Claude as a fallback — deterministic only if Gemma unavailable
